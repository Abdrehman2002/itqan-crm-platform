# Bug Report: Malformed UUID in path returns 500 instead of 400

## Severity
Low

## Environment
- API: http://129.121.115.99:3000 (production)
- Tested: 2026-06-19 21:44 UTC

## Steps to Reproduce
1. Login as tenant_admin and obtain JWT.
2. GET `/api/v1/contacts/not-a-uuid` with `Authorization: Bearer <token>`.

## Expected Behavior
HTTP 400 with a validation error message, since `not-a-uuid` is not a valid UUID — same format used by other validation failures (`{"success":false,"error":{"code":"VALIDATION_ERROR",...}}`).

## Actual Behavior
HTTP 500 with `{"success":false,"error":{"code":"INTERNAL_ERROR","message":"An unexpected error occurred"}}`. The Postgres "invalid input syntax for type uuid" error leaks through as an unhandled 500 instead of being caught as a validation error.

## Evidence
```
$ curl -s "http://129.121.115.99:3000/api/v1/contacts/not-a-uuid" -H "Authorization: Bearer $ADMIN" -w "\nHTTP %{http_code}\n"
{"success":false,"error":{"code":"INTERNAL_ERROR","message":"An unexpected error occurred"}}
HTTP 500
```

Compare: `/api/v1/contacts?pageSize=100000` correctly returns 400 with `VALIDATION_ERROR`. Path params should get the same Zod UUID validation that the query params get.

Fix: validate path params with `z.string().uuid()` in the route handler before hitting the DB, or add a 400-mapping for Postgres error code `22P02` (invalid text representation).

## Related
- Endpoint: `GET /api/v1/contacts/:id` (likely affects other entity GET-by-id endpoints too — `/companies/:id`, `/deals/:id`, `/tickets/:id`, etc.)

### FAILED — see Bug Reports
- **BUG-001 (Critical):** `/api/v1/opportunities` (GET list and GET by id) returns 500. Root cause: SQL references `c.name` on `contacts` table, which has only `first_name`/`last_name`. The entire Sales Opportunities feature is unusable. File: `packages/api/src/routes/opportunities.ts` lines 87, 90, 113, 117.
- **BUG-002 (High):** `/api/v1/sales/templates` returns the raw pg `QueryResult` object as `data` instead of the rows array. Frontend list of invoice templates will break. File: `packages/api/src/routes/sales/invoice-templates.ts` line 57–65.
- **BUG-003 (Low):** Malformed UUID in path (`/api/v1/contacts/not-a-uuid`) returns 500 instead of 400. Likely affects every `:id` route that doesn't pre-validate the UUID. Postgres `22P02` should map to a `VALIDATION_ERROR` response.

## Notes / Observations (not bugs, worth knowing)
- All 4 agent/manager test accounts have `manager_id = NULL`. The manager hierarchy can't be meaningfully tested until at least one agent reports to hassan.
- The 12 existing tickets are in a legacy `General Support` queue with `department_type = NULL`. New voice-agent–generated tickets should flow into the dept queues — recommend seeding 1 ticket per dept queue and re-running isolation tests.
- The `pending_or_failing` webhook count is 0 only because no webhooks are configured; the queue logic itself was not exercised end-to-end.
- The spec used `workspaceSlug` for login; the API expects `tenantSlug`. Worth aligning spec/docs to avoid future confusion.
- Heartbeat is at `/auth/heartbeat`, not `/api/v1/auth/heartbeat` (spec was wrong).

## Recommendation

Once BUG-001 and BUG-002 are patched and redeployed, this can be re-tested in ~10 minutes.
